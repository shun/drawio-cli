# drawio-cli Design Document

## 1. 概要 (Overview)

`drawio-cli` は、draw.io の公式描画エンジンをローカル環境で動かし、コマンドラインからダイアグラムの生成（SVGエクスポート）や編集を行うための自己完結型 CLI ツールです。

本ツールは、LLM（AIエージェント）と人間がシームレスにダイアグラムを共同編集・管理できる新しいドキュメントワークフローを実現することを目的としています。

### 主なユースケース
1. **LLMによる自動生成 (`export`)**: LLM が出力した XML ベースの `.drawio` ファイルから、Markdown に直接埋め込める美しい `.drawio.svg` をヘッドレスで自動生成する。
2. **人間による直感的な編集 (`edit`)**: 生成された `.drawio.svg` を、ローカルブラウザ（または VSCode拡張機能）を使って GUI 上で手軽に微調整・上書き保存する。

## 2. アーキテクチャ (Architecture)

本ツールは、draw.io 公式リポジトリの `webapp` 資産から「エクスポート」と「エディタ」のコア機能に必要なファイルのみを抽出し、バンドルしています。（約160MBから約58MBへ劇的に削減済）

### 2.1. 構成コンポーネント

* **抽出・ビルドスクリプト (`scripts/copy-drawio.js`)**
  * 公式リポジトリから CLI の動作に必要なファイルのみを厳選するハイブリッド方式のフィルタリング（`js` フォルダのホワイトリスト化と、`stencils` 等のブラックリスト化）を実行します。
* **Exporter モジュール (`src/exporter/index.ts`)**
  * Playwright (Chromium) をバックグラウンドで起動し、バンドルされた `export3.html` を読み込みます。
  * `page.evaluate()` を通じて XML データを注入し、draw.io 本体のエンジン (`app.min.js`) に図形とフォントの計算・レンダリングを行わせます。
  * Electron 環境をモックすることで、完成した SVG データ (`content="..."` 属性付き) を Node.js 側へ吸い出し、ファイルに保存します。
* **Server モジュール (`src/server/index.ts`)**
  * `edit` コマンド実行時に Express サーバーを立ち上げます。
  * バンドルされた `index.html?embed=1&modify=1` を iframe 内で開き、ローカルの `.drawio.svg` ファイルと `postMessage` 経由でデータをやり取りします。
  * ユーザーがブラウザ上で「保存」を実行すると、その XML データを即座にローカルの SVG ファイルへ上書きエクスポートします。

### 2.2. なぜヘッドレスブラウザ (Playwright) が必要なのか？

draw.io の描画エンジンは、DOM API（`document.createElement` 等）やブラウザのフォントレンダリング機構（文字幅のピクセル計算やワードラップ）に完全に依存しています。
JSDOM 等の擬似的な環境では正確なレイアウト計算（特にテキストの折り返しや MathJax 等）が不可能なため、「本物のブラウザ環境を裏で動かして描画させる」アプローチが最も正確かつ唯一の手段となります。

## 3. LLMと人間のハイブリッド・ワークフロー

このツールが提供する最大の価値は、**「構造（XML）」と「表現（SVG）」の分離と統合** にあります。

### LLMにとってのメリット
* LLM は複雑なベクターパス（SVG）を自力で計算することはできません。
* しかし、論理的なつながりを持つ XML (`.drawio`) を書くことは得意です。
* `drawio-cli export` を使えば、自分が書いた XML を人間が見て美しいと感じる完璧な SVG 画像に一発で変換できます。

### ワークフローの全体像

1. **AIのターン (生成)**
   LLM はユーザーの指示（例：「システム構成図を書いて」）を受け、`architecture.drawio` (純粋なXML) を生成します。
   その後、CLI を実行して SVG に変換します。
   ```bash
   drawio-cli export architecture.drawio -o architecture.drawio.svg
   ```
2. **AIのターン (後処理)**
   生成が完了したら、元となった `architecture.drawio` は削除するか、または残しておきます（どちらでも可）。最終的な成果物としては、画像でありながらデータを内包する `architecture.drawio.svg` がリポジトリにコミットされます。
3. **人間のターン (プレビューと微調整)**
   人間は GitHub や Markdown 上で、その SVG を即座に画像としてプレビューできます。
   もし色や配置を少し直したい場合は、CLI の編集コマンドを叩きます。
   ```bash
   drawio-cli edit architecture.drawio.svg
   ```
   ブラウザが開き、直感的な GUI でマウス操作による修正が可能になります。保存すると自動でファイルが更新されます。
   *(※人間が VSCode の Draw.io Integration 拡張機能をインストールしている場合は、エディタ上でそのままファイルを開いて編集することも可能です。)*

## 4. データフォーマット戦略 (Pros/Cons)

LLM が作成するファイルの拡張子についての設計判断です。

### 4.1. `.drawio` (XMLファイル) を入力とする理由
* **意図の明確さ**: 単なる `.xml` ではなく `.drawio` とすることで、それがダイアグラムのソースコードであることを人間や IDE（VSCode等）に明確に伝達できます。
* **LLMによる修正の容易さ**: ファイル置換ツール等で XML の特定ノードを直接書き換えることが容易です。

### 4.2. 最終成果物を `.drawio.svg` (XML内包SVG) とする理由
* **Single Source of Truth**: 画像と編集用ファイルが分かれていると生じる「二重管理（どちらかが古くなる問題）」を完全に防げます。
* **即時プレビュー**: Markdown ドキュメントにそのまま `![図](architecture.drawio.svg)` として埋め込めるため、ドキュメントの可読性が飛躍的に向上します。
* **編集の復元**: SVG 内の `<svg content="...">` 属性に完全な XML がエンコードされて埋め込まれているため、いつでも元の編集状態を完全に復元できます。

## 5. AgentSkill の設計

LLM（Gemini 等）が自律的にこのツールを使いこなすための `drawio-cli-skill` を実装しています。

### スキルの特徴
* **Mermaid の代替としてプロンプト**: Mermaid では表現力が不足するアーキテクチャ図やフローチャートにおいて、積極的にこのスキルを使うようプロンプト（`SKILL.md`）で指示しています。
* **リファレンスの同梱**: LLM がゼロから正しい XML を書けるよう、ノードやエッジの基本構造、カラーコード、座標計算のヒントをまとめた `drawio-xml-basics.md` を同梱し、プログレッシブディスロージャーの原則で必要な時だけ読み込ませます。
* **人間へのガイダンス**: LLM が図を作成し終えた際、人間に対して「VSCodeの拡張機能や CLI の `edit` コマンドで簡単に手直しできます」と案内するよう設計されています。これにより、AIが80%を作り、人間が残り20%のレイアウトを整えるという理想的な体験を誘導します。
